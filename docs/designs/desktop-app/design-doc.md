## Problem Statement

Agent of Empires has a working web dashboard (`aoe serve`) with live terminal streaming, REST API, and token auth. Power users love it. But to reach non-technical users who are "immediately scared away from the command line," aoe needs a native macOS app they can download and double-click. No terminal, no `brew install`, no tmux knowledge.

The second problem: you can't check on your agents from your phone. The web dashboard works in mobile browsers, but getting the URL + auth token onto your phone is friction. The vision: a menu bar app on your Mac that shows a QR code. You scan it. Your phone opens the dashboard. You're managing 6 AI agents from your couch in under 30 seconds.

## What Makes This Cool

Download a .dmg. Drag to Applications. Open it. A native macOS window shows your AI agent sessions with live terminal streaming. All 9+ supported agents, Docker sandboxing, git worktrees, everything aoe does, but in a window instead of a terminal.

Then you notice the menu bar icon. Click it. Toggle "Remote Access." A QR code appears. You scan it with your phone. Safari opens and you're looking at the same dashboard, live terminals scrolling. Tap into a session. Your agent is waiting for approval. You tap approve. It continues coding. You put your phone down and go back to dinner.

That's the "whoa" moment: Conductor Build can't do this. Codex App can't do this. Astropad Workbench requires a whole different setup. With aoe, it's scan and you're in.

## Constraints

- Must reuse the existing web dashboard (React + axum backend), not build a separate native UI
- Must ship as a single .app bundle (no external dependencies for the end user)
- tmux must be bundled if feasible; Homebrew auto-install is the fallback if static compilation proves problematic (the .app user shouldn't know tmux exists either way)
- Must support Apple Silicon and Intel Macs (universal binary)
- Requires Apple Developer account for code signing + notarization ($99/year)
- The CLI experience (`aoe` in the terminal) must continue to work alongside the desktop app
- Remote access toggle must be a deliberate user action with clear security implications
- LAN-only remote access for v1 (internet tunneling is documented, not built-in)
- v1 targets macOS only. Windows (via Tauri) and Linux (AppImage) are deferred to v2; Tauri's cross-platform support makes this a natural extension, not a rewrite.

## Premises

1. **Tauri 2.0 is the right framework.** Rust-native, uses macOS system WKWebView, ships small binaries (~15MB), has tray/menu bar support. The alternative (pure Swift) locks to macOS and requires maintaining two languages.
2. **The phone is the primary *remote* client; the desktop window is the primary *local* client.** The .app shows the full dashboard in a native window AND manages server/remote access from the menu bar.
3. **Non-technical users will download a .dmg, not run `cargo install`.** Code signing, notarization, Homebrew Cask, and a website download page are non-optional for v1.
4. **LAN-only remote access is enough for v1.** Internet access via Tailscale/Cloudflare Tunnel is documented, not built-in. Building a tunnel service is an ocean.

## Approaches Considered

### Approach A: Tauri Full Desktop App

A `desktop/` workspace member. Tauri opens a native macOS window (WKWebView) loading the web dashboard. System tray icon with session count, remote toggle, and QR code. Server starts automatically, binds to 0.0.0.0 with middleware blocking non-local connections until remote is toggled.

Effort: L (human: ~1 month / CC: ~4-6 hours)
Risk: Med
Completeness: 7/10

### Approach B: Tauri Menu Bar Only (no window)

Same Tauri setup but NO main window. Lives entirely in the menu bar. "Open Dashboard" launches the system browser. The .app is purely a server manager.

Effort: M (human: ~2 weeks / CC: ~2-3 hours)
Risk: Low-Med
Completeness: 5/10

### Approach C: Tauri Full + QR Code Pairing Flow (CHOSEN)

Approach A plus a polished pairing experience: QR code in a native tray popover, auth pre-baked into the URL, PWA install prompt on phone, and a first-launch wizard for non-technical users.

Effort: L-XL (human: ~5 weeks / CC: ~6-8 hours)
Risk: Med
Completeness: 9/10

## Recommended Approach

**Approach C: Tauri Full Desktop App with QR Code Pairing.**

The QR pairing flow IS the product differentiator. Without it, the app is Conductor Build with a remote toggle. With it, it's a "scan from your phone and you're managing agents" experience that no competitor offers.

### Architecture

```
.app bundle (Agent of Empires.app)
  |
  |-- Tauri shell (native macOS window + WKWebView)
  |     |-- Loads http://127.0.0.1:<port>/?token=<token>
  |     |-- Full web dashboard: sessions, terminals, diffs, settings
  |
  |-- Menu bar tray icon
  |     |-- Session count badge (green/yellow/red by status)
  |     |-- "Remote Access: Off" toggle
  |     |-- When toggled on: QR code popover + LAN URL + auth token
  |     |-- "Open Dashboard" (focuses the native window)
  |     |-- "Quit" (stops server, exits app; tmux sessions persist)
  |
  |-- Rust backend (same library, Tauri entrypoint)
  |     |-- agent_of_empires::server::start_server() on 127.0.0.1
  |     |-- AtomicBool middleware: rejects non-local when remote is off
  |     |-- All existing aoe logic: sessions, tmux, containers, worktrees
  |
  |-- Bundled tmux (static binary in .app/Contents/MacOS/)
  |-- Bundled web assets (embedded via rust-embed at compile time)
```

### Tauri entrypoint (`desktop/src/main.rs`)

Note: the code below is pseudocode illustrating the architecture, not a compilable listing.

```rust
#[tokio::main]
async fn main() {
    let port = find_available_port();
    let token = generate_auth_token();
    let remote_enabled = Arc::new(AtomicBool::new(false));

    // Clone values needed by multiple closures before moving into setup
    let server_token = token.clone();
    let server_remote = remote_enabled.clone();
    let tray_token = token.clone();
    let tray_remote = remote_enabled.clone();

    tauri::Builder::default()
        .plugin(tauri_plugin_shell::init())
        .setup(move |app| {
            // Start the web server on the shared tokio runtime.
            // remote_enabled lives on AppState (see "New Rust library changes").
            tokio::spawn(async move {
                agent_of_empires::server::start_server_with_options(
                    port, server_token, server_remote
                ).await
            });

            // Create the main window pointing at the local dashboard
            let url = format!("http://127.0.0.1:{}/?token={}", port, tray_token);
            // Tauri webview window loads this URL

            // Set up system tray
            setup_tray(app, port, tray_token, tray_remote)?;

            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error running tauri app");
}
```

### Remote Access toggle: implementation detail

The server always binds to `0.0.0.0` on startup. A shared `Arc<AtomicBool>` controls whether non-local connections are accepted:

- **Off (default)**: Axum middleware checks `request.extensions().get::<ConnectInfo<SocketAddr>>()`. If not `127.0.0.1` or `::1`, returns 403. The local webview works normally.
- **On**: Middleware passes all authenticated connections. Tray updates to show QR code + LAN URL.

This avoids rebinding the TCP listener (which would kill active WebSocket sessions). The toggle is instant and non-disruptive.

### QR Code pairing flow

When the user toggles remote access on:

1. The app detects the machine's LAN IP address (e.g., `192.168.1.42`)
2. Constructs the URL: `http://192.168.1.42:<port>/?token=<token>`
3. Generates a QR code using the `qrcode` crate (renders to PNG in memory)
4. Displays the QR code in a small frameless Tauri window anchored below the tray icon (native NSMenu does not support embedded images; this uses Tauri's window positioning API to create a popover-style panel)
5. Also copies the URL to clipboard and shows it as text below the QR code
6. A macOS notification says: "Remote access enabled. Scan the QR code or open the URL on your phone."

On the phone side:
1. User scans QR code with camera
2. Safari opens the dashboard URL with token pre-baked
3. The auth cookie is set on first load
4. The web dashboard renders (already mobile-responsive)
5. If desired, user can "Add to Home Screen" (PWA manifest is already in place)

### Security model

- **Auth**: Always enabled. 32-character alphanumeric token generated on app launch.
- **Local webview**: Token pre-injected via URL parameter, sets HTTP-only cookie.
- **Remote clients**: Must have the token (via QR code URL or manual entry).
- **Scope**: LAN-only by default. 0.0.0.0 binding is blocked by most routers from internet. Users who want internet access are documented to use Tailscale, Cloudflare Tunnel, or SSH port forwarding.
- **TLS**: Not built-in for v1. The auth token travels in plaintext on the LAN. The QR code pairing UI displays a warning: "Only use remote access on trusted networks (home, office). Do not enable on public WiFi." Future: self-signed TLS certificate generated at app launch via rustls, which eliminates plaintext token exposure at the cost of a browser certificate warning on the phone.
- **Token rotation**: New token each app launch. No session expiry in v1.
- **Visual indicator**: The tray icon changes color when remote access is on (red dot or similar) so the user always knows.

### tmux bundling

The .app bundles a pre-built tmux binary at `.app/Contents/MacOS/tmux`. The Tauri entrypoint prepends this path to `$PATH` at startup.

**Build**: Compile tmux from source in CI (GitHub Actions macOS runner) targeting arm64 and x86_64, then `lipo` into a universal binary. Dependencies (ncurses, libevent) are statically linked. The tmux binary is ~3MB.

**Size budget (estimated, verify with spike)**: Tauri shell + aoe binary + tmux + web assets. Universal binaries (arm64 + x86_64) and vendored dependencies (git2, openssl) may push the total higher than expected. Target: under 30MB. The proof-of-concept spike (Next Steps #1) should measure actual binary sizes before committing to a hard ceiling.

**Known risks**: Statically linking ncurses on macOS has historical issues with terminfo database path resolution. The tmux spike must test on a clean macOS install without Homebrew to verify the bundled binary works standalone.

### First-launch experience (v1.1, deferred)

The web dashboard already has session creation UI. For v1, the "zero sessions" state shows the existing dashboard with its "No sessions" view plus a prominent "Create Session" button. This is sufficient for the core value proposition (native window + QR pairing).

A guided first-launch wizard (agent picker, folder picker, prerequisite check for agent CLIs) is deferred to v1.1 to keep v1 scope focused on the differentiating features.

### Auto-updates

Uses Tauri's built-in updater plugin. Checks GitHub Releases for new versions on a 24-hour interval. Shows a native macOS notification when an update is available. User clicks to download and install.

### Distribution

1. **GitHub Releases**: Ship `.dmg` alongside the existing CLI binary
2. **Homebrew Cask**: `brew install --cask agent-of-empires` (separate from CLI formula)
3. **Website**: Direct .dmg download from the docs site
4. **.app is separate from CLI**: Installing the .app does NOT replace `aoe` in the terminal. They coexist.

### Code signing + notarization

Required for macOS distribution. Handled via environment variables in CI:
- `APPLE_CERTIFICATE`, `APPLE_CERTIFICATE_PASSWORD`, `APPLE_SIGNING_IDENTITY`
- `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_TEAM_ID`

Added to GitHub Actions release workflow as a separate job.

### CLI and desktop coexistence

If both the CLI and desktop app are installed:
- `aoe serve` checks if the desktop app's server is already running (reads port from `~/.agent-of-empires/serve.url`). If yes, opens a browser to it instead of starting a new server.
- The desktop app and CLI share the same session storage (`~/.agent-of-empires/` or `~/.config/agent-of-empires/`).
- tmux sessions are shared; both can attach to the same sessions.

### New Rust library changes needed

1. **`src/server/mod.rs`**: Extract `start_server()` into a more flexible `start_server_with_options()` that accepts a config struct. Add `remote_enabled: AtomicBool` as a field on `AppState` (consistent with the existing pattern where all shared state lives on `AppState`). The desktop app sets this field; the CLI leaves it as `None` (always-local behavior unchanged).
2. **`src/server/middleware.rs`** (new): Remote access middleware that reads `AppState.remote_enabled` + `ConnectInfo` to allow/deny non-local requests.
3. **QR code**: Add `qrcode` crate as a dependency in the `desktop/` workspace member (not behind a feature flag in the main crate).

## Failure Modes and Edge Cases

| Scenario | Behavior |
|----------|----------|
| Port already in use | Try the next 10 ports (8081-8090). If all taken, show a native macOS alert: "Could not start server. Another application is using the required ports." |
| Bundled tmux fails to execute (Gatekeeper, permissions) | Detect the error, show alert suggesting the user right-click > Open to bypass Gatekeeper. If bundled tmux is missing entirely, fall back to system tmux (`which tmux`), then prompt to install via Homebrew. |
| LAN IP detection returns multiple interfaces or VPN | Show all detected LAN IPs in the QR popover and let the user pick. Default to the first non-VPN, non-Docker interface. |
| Server crashes mid-session | Tauri catches the panic/error. Tray icon turns gray. "Server stopped unexpectedly. Restart?" button in tray menu. tmux sessions survive (they're independent processes). |
| Agent CLI not installed | When user creates a session for an agent whose CLI is not on `$PATH`, show an error with a link to the agent's install instructions. Do not silently fail. |
| CLI and desktop port conflict | `aoe serve` checks `~/.agent-of-empires/serve.url` and `serve.pid` before starting. If the desktop app is already running, `aoe serve` prints the existing URL and opens a browser to it instead of starting a second server. |

## Open Questions

1. **Auto-launch on macOS login?** Deferred to v1.1. Docker Desktop does this because it provides system services; aoe does not need to run continuously. Focus v1 on the download-and-run experience.
2. **Multiple monitors**: Should the tray popover QR code also be displayable as a floating window for easier scanning?
3. **Token sharing**: If the desktop app and CLI share the same session storage, should they also share the same auth token? Or are they independent servers with independent tokens?
